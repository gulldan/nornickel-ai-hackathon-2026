# Фабрика R&D-гипотез

Решение команды Квадрицепс.

Фабрика R&D-гипотез помогает перейти от набора документов к портфелю проверяемых
идей для исследований и технологических проектов.

Пользователь задает KPI, то есть измеримый целевой показатель. Система ищет
факты в базе знаний, формулирует гипотезы, показывает источники, оценивает риск
и предлагает план проверки и запуска НИОКР.

Главный результат: не чат по PDF, а доска гипотез, которую эксперт может
проверить, отсортировать и принять в работу.

## Для кого

1. Эксперт R&D получает идеи с источниками, механизмом и планом эксперимента.
2. Руководитель видит портфель направлений, приоритеты, риски и готовность к
   проверке.
3. Куратор базы знаний загружает документы и видит, какие материалы уже готовы к
   поиску.
4. Администратор управляет пользователями, весами рейтинга, лимитами генерации и
   метриками качества.

## Что умеет система

1. Загружает PDF, офисные файлы, таблицы, письма, архивы, изображения и сканы.
2. Извлекает текст и структуру. Для сканов использует OCR, то есть распознавание
   текста на изображениях.
3. Делит документы на фрагменты и сохраняет координаты источников.
4. Ищет по смыслу и по точным словам.
5. Отвечает на вопросы с цитируемыми источниками.
6. Группирует документы по темам.
7. Предлагает KPI по базе знаний.
8. Генерирует гипотезы под KPI и ограничения пользователя.
9. Проверяет доводы в пользу гипотезы и доводы против нее.
10. Считает рейтинг по открытой формуле, а не по скрытому мнению модели.
11. Экспортирует портфель и отчеты по гипотезам.

## Что важно эксперту

1. У каждой гипотезы есть источник с точностью до фрагмента документа.
2. Гипотеза содержит управляемое изменение, ожидаемый механизм и измеримый
   результат.
3. Проверка ищет не только подтверждения, но и ограничения.
4. План эксперимента отделен от общего описания идеи.
5. Решение остается за экспертом. Его оценка влияет на следующий цикл работы.

## Что важно бизнесу

1. Система показывает не один ответ, а портфель вариантов.
2. Приоритет объясняется формулой из шести факторов.
3. Веса факторов можно менять под стратегию организации.
4. Метрики качества доступны в админке.
5. Контур можно запустить локально. В этом режиме документы не уходят во внешние
   сервисы.

## Как устроен рейтинг

Рейтинг считает код в
`backend/main-service/internal/application/scoring.go`.

```text
Score = 0.25 * соответствие KPI
      + 0.20 * доказательная база
      + 0.20 * новизна
      + 0.15 * ценность
      + 0.10 * управляемость риска
      + 0.10 * готовность технологии
```

На карточке гипотезы виден вклад каждого фактора. Эксперт может изменить веса в
`/admin/settings`.

## Как работает основной путь

1. Пользователь загружает документы.
2. Система сохраняет исходные файлы в локальное S3-совместимое хранилище.
3. Парсеры извлекают текст, таблицы и структуру.
4. Сканы проходят OCR через PaddleOCR-VL.
5. `chunk-splitter` делит текст на фрагменты.
6. `f2llm-service` строит смысловые векторы фрагментов.
7. Qdrant хранит векторы. OpenSearch хранит индекс точного поиска.
8. `llm-service` ищет источники, сортирует их реранкером и готовит ответ.
9. `main-service` сохраняет результат, источники, след поиска и события для
   интерфейса.
10. Фоновые воркеры строят темы, переносят сильные идеи между близкими KPI,
    ищут мосты между темами и считают метрики.

RAG означает поиск ответа по базе знаний с источниками. LLM означает большую
языковую модель.

## Быстрый запуск

Нужны Linux, `make`, Docker 24 или новее, Docker Compose v2, NVIDIA GPU и NVIDIA
Container Toolkit.

Запуск из корня репозитория:

```bash
make up
```

`make up` создает `infra/.env`, заполняет локальные секреты, собирает образы и
поднимает контур.

Адреса после запуска:

1. Интерфейс: `http://localhost:5173`.
2. API: `http://localhost:8080`.
3. RabbitMQ: `http://localhost:15672`.
4. Grafana: `http://localhost:3000`.
5. Prometheus: `http://localhost:9090`.

Первый вход: пользователь `admin`, пароль из `ADMIN_PASSWORD` в `infra/.env`.

Подробный запуск: [docs/DEPLOY-QUICKSTART.md](docs/DEPLOY-QUICKSTART.md).

## Полностью локальный режим

Эмбеддинги, реранкер и OCR уже работают локально в Docker Compose.

Генерация текста по умолчанию настраивается через `VLLM_URL`. Чтобы включить
локальную генерацию, положите GGUF-модель в `models/gguf/` и запустите:

```bash
docker compose -f infra/docker-compose.yml --profile local-llm up -d llama-server
```

Затем задайте в `infra/.env`:

```text
VLLM_URL=http://llama-server:8090
VLLM_API_KEY=
```

Если также выключить поиск открытых публикаций через `PUBSEARCH_ENABLED=false`,
рабочий контур не обращается к внешним сервисам.

## Проверка перед показом

```bash
make config
make build
make frontend-check
make demo-resilience
```

После этого проверьте путь в интерфейсе:

1. Загрузить документы.
2. Создать KPI.
3. Сгенерировать гипотезы.
4. Открыть карточку гипотезы.
5. Проверить источники, рейтинг и план эксперимента.
6. Задать один вопрос в поиске по базе знаний.

Сценарий демо: [docs/DEMO-CASE-TOF.md](docs/DEMO-CASE-TOF.md).

## Структура репозитория

1. `backend/`: Go-сервисы, авторизация, API, база данных, парсеры и RAG.
2. `frontend/`: пользовательский интерфейс на React и Vite.
3. `ml-services/`: ML-сервисы и фоновые воркеры.
4. `infra/`: Docker Compose, nginx, мониторинг и инфраструктурные настройки.
5. `docs/`: актуальная документация.
6. `datasets/`: демонстрационные и тестовые материалы.
7. `models/`: локальные модели, если они нужны для запуска без внешнего
   генератора.

## Основные команды

```bash
make up              # собрать и запустить весь контур
make down            # остановить контур
make config          # проверить docker-compose
make build           # собрать Go-сервисы
make test            # запустить Go-тесты
make frontend-check  # проверить фронтенд
make demo-resilience # проверить резервное поведение RAG
```

Для локальной разработки фронтенда:

```bash
make dev
```
