# Архитектура

Платформа состоит из сервисов приложения, ML-сервисов и инфраструктурных
хранилищ. Локальный запуск описан в `infra/docker-compose.yml`.

Публичные входы:

1. Пользовательский интерфейс на `http://localhost:5173`.
2. API через nginx на `http://localhost:8080`.

Служебные хранилища работают внутри Docker-сети.

## Основные сервисы

1. `nginx`: единый вход для API и WebSocket.
2. `main-service`: HTTP API, права доступа, документы, чат, KPI и гипотезы.
3. `auth`: вход пользователей и JWT, то есть токены доступа.
4. `db-service`: единственный владелец схемы Postgres.
5. `llm-service`: RAG-поиск, сортировка источников и генерация ответа.
6. `pdf-parser`, `office-parser`, `email-parser`, `db-parser`: извлечение текста
   из разных форматов.
7. `archive-worker` и `archive-scan`: обработка архивов.
8. `ocr-service`: адаптер распознавания текста на изображениях.
9. `chunk-splitter`: делит текст на фрагменты, получает эмбеддинги и пишет
   индексы.
10. `f2llm-service`: локальная модель эмбеддингов.
11. `reranker-service`: локальная модель точной сортировки источников.
12. `paddleocr-vl-service`: OCR-движок для сканов и изображений.
13. `graph-compute`: графовый расчет тем и мостов между темами.
14. `cluster-worker`: обновляет доску тем.
15. `discovery-worker`: переносит сильные гипотезы между близкими KPI. В режиме
    `bridge` создает гипотезы на стыке тем.
16. `raptor-worker`: строит обзорные узлы для широких вопросов.
17. `eval-service`: считает метрики качества.
18. `itc-worker`: считает индекс технологичности.

## Хранилища

1. Postgres хранит пользователей, документы, KPI, гипотезы, историю и настройки.
2. Qdrant хранит смысловые векторы фрагментов и обзорных узлов.
3. OpenSearch хранит индекс точного поиска по словам.
4. SeaweedFS работает как локальное S3-совместимое хранилище файлов.
5. RabbitMQ передает фоновые задачи.
6. Valkey хранит кэш, версии корпуса и сигналы для фоновых воркеров.
7. Prometheus и Grafana отвечают за метрики.
8. Vector включается отдельным профилем для сбора логов.

## Загрузка документа

1. Пользователь загружает файл в интерфейсе.
2. `main-service` сохраняет исходный файл в SeaweedFS.
3. `main-service` создает запись документа через `db-service`.
4. `main-service` публикует задачу в RabbitMQ.
5. Парсер извлекает текст. Для скана задача уходит в `ocr-service`.
6. `ocr-service` вызывает `paddleocr-vl-service`.
7. `chunk-splitter` делит текст на фрагменты.
8. `chunk-splitter` вызывает `f2llm-service` и получает векторы.
9. Фрагменты записываются в Qdrant и OpenSearch.
10. Статус документа меняется на indexed.
11. Интерфейс получает обновление через WebSocket.

## Ответ на вопрос

1. Пользователь задает вопрос.
2. `main-service` передает запрос в `llm-service`.
3. `llm-service` строит смысловой вектор вопроса через `f2llm-service`.
4. Qdrant ищет близкие по смыслу фрагменты.
5. OpenSearch ищет точные совпадения слов, формул и обозначений.
6. `llm-service` объединяет результаты через RRF, то есть объединение рангов.
7. `reranker-service` точнее сортирует найденные фрагменты.
8. Если найден обзорный узел RAPTOR, `llm-service` раскрывает его в реальные
   исходные фрагменты.
9. Генератор получает только найденный контекст.
10. Ответ возвращается с источниками и следом поиска.
11. История вопроса и ответа сохраняется в Postgres.

Если генератор недоступен, а источники найдены, чат возвращает короткий ответ из
найденных фрагментов. Такой результат не кэшируется.

## Генерация гипотез

1. Пользователь задает KPI и ограничения.
2. `main-service` собирает контекст из базы знаний и приложенных документов.
3. `llm-service` находит релевантные источники.
4. Генератор формулирует гипотезы в структурированном JSON.
5. `main-service` привязывает источники и считает рейтинг.
6. Автопроверка ищет подтверждения и ограничения.
7. Обогащение добавляет механизм, причинную цепочку и план эксперимента.
8. Порог качества архивирует слабые или неполные гипотезы.
9. Эксперт принимает, отклоняет или редактирует гипотезу.

Подробности: [product/HYPOTHESIS_GENERATION.md](product/HYPOTHESIS_GENERATION.md).

## Темы и мосты

1. После индексации документа `main-service` обновляет версию корпуса в Valkey.
2. `cluster-worker` ждет паузу в загрузках и вызывает `graph-compute`.
3. `graph-compute` читает векторы из Qdrant и группирует документы в темы.
4. `cluster-worker` публикует темы через API.
5. `discovery-worker` по умолчанию работает в режиме `transfer`.
6. В режиме `transfer` он переносит сильную гипотезу с одного KPI на близкий KPI,
   если у второй цели мало покрытия.
7. В режиме `bridge` он вызывает `graph-compute` для поиска мостов между темами.
8. Мост проходит проверку на источники, новизну и несколько независимых сигналов.
9. После проверки мост может стать гипотезой.

## RAPTOR

RAPTOR здесь означает слой обзорных узлов над обычными фрагментами.

По умолчанию включен режим `RAPTOR_MODE=incremental`. В этом режиме
`raptor-worker` создает стабильный краткий обзор для каждого нового или
измененного документа.

Обзорный узел помогает широким вопросам. В ответе пользователю все равно
показываются реальные исходные фрагменты, а не синтетический краткий обзор.
