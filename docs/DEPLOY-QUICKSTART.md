# Быстрый запуск

Документ описывает запуск локального стенда для проверки и демонстрации.

Контур поднимается через Docker Compose. Основной вход для пользователя:
`http://localhost:5173`.

## Требования

1. Linux.
2. `make`.
3. Docker 24 или новее.
4. Docker Compose v2.
5. NVIDIA GPU.
6. NVIDIA Container Toolkit. Команда `docker info` должна показывать среду
   выполнения `nvidia`.
7. Около 25 ГБ свободного места под образы и веса моделей.
8. Доступ в интернет при первом старте, если веса моделей еще не скачаны.

Базовый контур использует около 7 ГБ видеопамяти для эмбеддингов, реранкера и
OCR. Локальная генерация через профиль `local-llm` требует дополнительной
видеопамяти. Точное значение зависит от GGUF-модели.

## Запуск

Из корня репозитория:

```bash
make up
```

Команда выполняет четыре действия:

1. Проверяет Docker и среду выполнения NVIDIA.
2. Создает `infra/.env` из `infra/.env.example`, если файла еще нет.
3. Заполняет пустые локальные секреты.
4. Собирает и запускает сервисы.

Если `infra/.env` уже существует, команда не меняет заполненные пароли и ключи.

## Адреса

1. Интерфейс: `http://localhost:5173`.
2. API: `http://localhost:8080`.
3. RabbitMQ: `http://localhost:15672`.
4. Grafana: `http://localhost:3000`.
5. Prometheus: `http://localhost:9090`.

Логин администратора: `admin`.

Пароль: значение `ADMIN_PASSWORD` в `infra/.env`.

## Проверка состояния

```bash
docker compose -f infra/docker-compose.yml ps
```

На первом запуске часть сервисов может ждать загрузку весов моделей. Обычно это
занимает несколько минут.

## Локальная генерация

Эмбеддинги, реранкер и OCR уже работают локально.

Для локальной генерации положите GGUF-модель в `models/gguf/` и запустите:

```bash
docker compose -f infra/docker-compose.yml --profile local-llm up -d llama-server
```

Задайте в `infra/.env`:

```text
VLLM_URL=http://llama-server:8090
VLLM_API_KEY=
```

После изменения `infra/.env` перезапустите сервисы, которым нужны новые
значения.

## Внешняя генерация

Контур поддерживает OpenAI-совместимый адрес генерации.

Основные переменные:

1. `VLLM_URL`: базовый адрес без `/v1`.
2. `VLLM_MODEL`: имя модели.
3. `VLLM_API_KEY`: ключ доступа, если он нужен провайдеру.
4. `VLLM_RPM`: мягкий лимит запросов в минуту.

Поиск открытых публикаций управляется переменной `PUBSEARCH_ENABLED`. Если
поставить `false`, система не будет обращаться к OpenAlex и Crossref.

## Безопасность стенда

1. Не коммитьте `infra/.env`.
2. Не коммитьте временные `.env.*` backup-файлы.
3. Перед внешним доступом проверьте `JWT_SECRET`, `ADMIN_PASSWORD`,
   `GRAFANA_ADMIN_PASSWORD`, `RABBITMQ_PASSWORD`, `POSTGRES_PASSWORD`,
   `S3_ACCESS_KEY` и `S3_SECRET_KEY`.
4. Публичная саморегистрация закрыта по умолчанию:
   `REGISTRATION_ADMIN_ONLY=true`.
5. Служебные базы данных не публикуются на хост. Доступ к ним нужен через
   `docker compose exec`.
6. Локальный S3-адрес `http://localhost:4566` подходит только для браузера на той
   же машине. Для удаленного браузера настройте внешний защищенный адрес и
   обновите `S3_PUBLIC_ENDPOINT`.

## Сценарий проверки

1. Откройте `http://localhost:5173`.
2. Войдите как `admin`.
3. Откройте раздел документов.
4. Загрузите PDF, DOCX, XLSX или изображение.
5. Дождитесь завершения обработки.
6. Создайте KPI в разделе целей.
7. Запустите генерацию гипотез.
8. Откройте карточку гипотезы.
9. Проверьте источники, рейтинг, ограничения и план эксперимента.
10. Задайте вопрос в поиске и проверьте ответ с источниками.

## Команды

```bash
make up              # запуск контура
make down            # остановка
make config          # проверка docker-compose
make build           # сборка Go-сервисов
make test            # Go-тесты
make frontend-check  # проверки фронтенда
make demo-resilience # проверки резервного поведения RAG
```

## Передача исходников

Для передачи исходников после финального коммита используйте архив:

```bash
mkdir -p out
git archive --worktree-attributes --format=tar.gz \
  -o out/rag-platform-organizers.tar.gz HEAD
```

`.gitattributes` исключает локальные инженерные каталоги и секреты из архива.
