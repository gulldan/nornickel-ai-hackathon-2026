# Алгоритмы и модели

Документ объясняет алгоритмы платформы простым языком. Внутренние протоколы и
имена сервисов вынесены в [../ARCHITECTURE.md](../ARCHITECTURE.md).

## Общая цепочка

Платформа проходит путь:

1. Материалы.
2. Смысловые фрагменты.
3. Поиск с источниками.
4. Темы.
5. KPI.
6. Гипотезы.
7. Приоритет.
8. Решение эксперта.

## Подготовка базы знаний

Задача: превратить разные файлы в единый корпус для поиска.

Система делает:

1. Извлекает текст, таблицы и структуру.
2. Распознает сканы и изображения.
3. Делит материал на фрагменты.
4. Сохраняет место фрагмента в документе.
5. Строит смысловое представление фрагмента.

Если важное число или таблица потеряны на этом этапе, честная гипотеза не сможет
на них опереться.

## Смысловые фрагменты

Документ не ищется целиком. Он делится на фрагменты, потому что вопрос обычно
относится к конкретному месту.

Для каждого фрагмента есть две версии:

1. Чистый текст для пользователя.
2. Контекстная версия для поиска.

Контекстная версия включает название документа и раздел. Это помогает короткому
фрагменту не потерять смысл.

## Смысловое представление

Embedding, то есть смысловой вектор, переводит текст в набор чисел.

Близкие по смыслу фрагменты получают близкие векторы.

Польза: система находит близкие идеи даже при разных словах и языках.

## Поиск по смыслу и по словам

Один способ поиска недостаточен.

Смысловой поиск полезен для идей и механизмов.

Точный поиск полезен для:

1. Марок материалов.
2. Формул.
3. Чисел.
4. Номеров стандартов.
5. Редких сокращений.

Система объединяет оба результата по рангам. Поэтому источник, найденный двумя
способами, получает преимущество.

## Точная сортировка источников

После первичного поиска кандидатов слишком много.

Реранкер сравнивает вопрос и каждый фрагмент вместе. Он точнее понимает, какой
фрагмент действительно отвечает на вопрос.

Система также ограничивает число фрагментов из одного документа. Это снижает
риск, что один длинный PDF вытеснит остальные источники.

## Ответ по базе знаний

Ответ создается только после выбора источников.

Генератор получает вопрос и ограниченный набор фрагментов. Он не выбирает корпус
заново.

Если источников нет, система должна отказаться от ответа. Это лучше, чем
выдуманный вывод.

## Обзорные узлы

Широкий вопрос может относиться к целому документу. Для этого система строит
обзорные узлы RAPTOR.

Обзорный узел помогает найти нужную область. Но в ответе пользователь видит
реальные исходные фрагменты.

## Темы корпуса

Темы помогают увидеть исследовательский ландшафт.

Система:

1. Собирает представление каждого документа из его фрагментов.
2. Соединяет близкие документы в граф.
3. Находит сообщества в графе.
4. Публикует темы в интерфейсе.

Темы показывают, где уже есть экспертиза, а где корпус слабый.

## Мосты между темами

Новые идеи часто появляются на стыке областей.

Система ищет пары тем, которые близки по смыслу, но редко связаны напрямую.

Для моста нужны:

1. Близость тем.
2. Слабая прямая связь.
3. Документы-посредники.
4. Несколько независимых сигналов.
5. Реальные источники.

Если мост проходит проверки, он может стать гипотезой.

## Проверяемая гипотеза

Хорошая гипотеза содержит:

1. Объект исследования.
2. Управляемое изменение.
3. Ожидаемый механизм.
4. Измеримый результат.
5. Условия проверки.
6. Источники в пользу.
7. Ограничения или доводы против.

Такую гипотезу можно обсуждать как план эксперимента, а не как общую идею.

## Проверка гипотезы

Проверка всегда ищет две стороны:

1. Подтверждения.
2. Ограничения и контраргументы.

Возможные вердикты:

1. Поддержана.
2. Опровергнута.
3. Смешанная.
4. Недостаточно данных.

Для R&D ранняя видимость ограничений экономит время и бюджет.

## Прозрачный приоритет

Приоритет считается по формуле.

Факторы:

1. Соответствие KPI.
2. Доказательная база.
3. Новизна.
4. Ожидаемая ценность.
5. Управляемость риска.
6. Готовность к проверке.

Формула разделяет ценность и риск. Новая идея может быть ценной, но еще слишком
дорогой или рискованной для ближайшего эксперимента.

## Уровень готовности

Уровень готовности технологии показывает, насколько идея близка к практике.

Низкий уровень не означает плохую идею. Он означает раннюю стадию исследования.

## Индекс технологичности

Индекс технологичности оценивает тему по наблюдаемым признакам:

1. Масштаб темы.
2. Новизна.
3. Влияние внутри корпуса.
4. Ресурсный и организационный след.

Индекс помогает сравнивать темы, а не отдельные красивые формулировки.

## Оценка качества

Качество нужно измерять регулярно.

Проверяются четыре группы:

1. Качество поиска.
2. Качество ответа.
3. Качество гипотез.
4. Качество межтематических связей.

Хорошая версия фабрики должна выполнять пять условий:

1. Источники можно открыть.
2. Гипотеза имеет критерий успеха.
3. Приоритет объясняется формулой.
4. Новые идеи проходят защиту от пересказа и завышенных обещаний.
5. Качество измеряется на контрольных примерах.

## Типы моделей

1. Модель распознавания текста извлекает текст из сканов и изображений.
2. Модель эмбеддингов строит смысловые векторы.
3. Реранкер точнее сортирует найденные источники.
4. LLM формулирует ответы, гипотезы и краткие обзоры.
5. Расчетные модели считают готовность, технологичность, новизну и приоритет.

LLM не является единственным источником истины. Все важные решения должны иметь
источники, формулы или явные правила.
