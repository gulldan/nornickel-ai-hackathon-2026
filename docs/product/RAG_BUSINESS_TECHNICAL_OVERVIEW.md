# RAG: поиск и ответ с источниками

RAG означает ответ по базе знаний с опорой на найденные источники.

Цель контура: дать пользователю ответ, который можно проверить по исходным
документам.

## Коротко

1. Документы превращаются в фрагменты.
2. Каждый фрагмент получает текст, координаты и смысловой вектор.
3. Вопрос пользователя ищется двумя способами: по смыслу и по точным словам.
4. Результаты объединяются по рангам.
5. Реранкер точнее сортирует найденные фрагменты.
6. Генератор получает только выбранные источники.
7. Ответ возвращается с цитатами и следом поиска.
8. История ответа сохраняется в Postgres.

## Индексация корпуса

При загрузке документа система делает следующие шаги:

1. Сохраняет исходный файл в SeaweedFS.
2. Создает запись документа в Postgres.
3. Извлекает текст через парсер.
4. Для скана вызывает OCR.
5. Делит текст на фрагменты.
6. Добавляет к фрагменту название документа и раздел для поиска.
7. Получает смысловой вектор.
8. Пишет вектор в Qdrant.
9. Пишет текстовую запись в OpenSearch.
10. Ставит документу статус indexed.

Пользователю показывается сырой фрагмент из документа. Контекстная версия нужна
только для поиска.

## Что хранится у фрагмента

1. `chunk_id`: идентификатор фрагмента.
2. `document_id`: идентификатор документа.
3. `filename`: имя файла.
4. `owner_id`: владелец.
5. `text`: текст для ответа и показа.
6. `chunk_index`: номер фрагмента в документе.
7. `char_start` и `char_end`: позиция в извлеченном тексте.
8. `page_start` и `page_end`: страницы, если парсер их знает.
9. `section_heading`: раздел.
10. `node_type`: обычный фрагмент или обзорный узел RAPTOR.
11. `member_ids`: исходные фрагменты обзорного узла.

## Поиск по смыслу

Смысловой поиск находит близкие идеи, даже если слова отличаются.

Пример: вопрос про коррозионную стойкость может найти фрагмент, где написано
corrosion resistance или passivation behavior.

За это отвечает `f2llm-service`. Модель задается переменными
`EMBEDDINGS_MODEL` и `F2LLM_MODEL`.

## Точный поиск

OpenSearch ищет точные слова, формулы, марки материалов, номера стандартов и
числа.

Это важно, потому что смысловая модель может сгладить важную точность.

Примеры точных запросов:

1. `Al-Mg-Si`.
2. `0,1 мм/год`.
3. Номер стандарта.
4. Марка реагента.

## Объединение результатов

У смыслового поиска и точного поиска разные шкалы. Их нельзя просто сложить.

Поэтому система объединяет места в выдаче. Этот метод называется RRF.

Смысл для пользователя простой: фрагмент, который хорошо найден двумя способами,
получает преимущество. Сильный результат одного способа тоже может попасть в
ответ.

## Реранкер

Реранкер сравнивает вопрос и каждый найденный фрагмент вместе. Это точнее, чем
первичный поиск, но дороже.

Поэтому реранкер применяется только к короткому списку кандидатов.

В локальном контуре работает `reranker-service`. Модель задается через
`RERANKER_MODEL` и `RERANKER_SERVICE_MODEL`.

Порог слабой доказательной базы задает `RAG_SCORE_FLOOR`. При смене реранкера
этот порог нужно заново проверить на eval-наборе.

## RAPTOR

RAPTOR добавляет обзорные узлы к обычным фрагментам.

Обзорный узел полезен для широкого вопроса, который относится не к одной строке,
а к целому документу или группе фрагментов.

Важное правило: пользователь не цитирует обзорный узел. Если поиск нашел такой
узел, `llm-service` раскрывает его в реальные исходные фрагменты и цитирует их.

По умолчанию используется `RAPTOR_MODE=incremental`.

## Генерация ответа

Генератор получает:

1. Вопрос пользователя.
2. Нумерованный список найденных фрагментов.
3. Инструкцию отвечать только по этим источникам.

LLM формулирует текст, но не выбирает источники заново.

Если источник не был передан в контекст, модель не должна на него ссылаться.

## Проверка цитат

Перед отправкой ответа система проверяет маркеры источников.

1. Найденные фрагменты нумеруются.
2. Модель пишет ответ с маркерами источников.
3. Система проверяет, что каждый маркер существует.
4. Несуществующие маркеры удаляются.
5. Интерфейс показывает кликабельные источники.

Так пользователь не видит ссылку на источник, которого не было в выдаче.

## Отказ от слабого ответа

Система не обязана отвечать любой ценой.

Ответ не генерируется, если:

1. Источники не найдены.
2. Реранкер включен, но лучший score ниже `RAG_SCORE_FLOOR`.

В этом случае пользователь получает честное сообщение о том, что релевантного
контекста недостаточно.

## Резервное поведение

Если часть контура недоступна, система старается сохранить полезный результат.

1. Если недоступен Qdrant, используется точный поиск.
2. Если недоступен OpenSearch, используется смысловой поиск.
3. Если недоступен реранкер, сохраняется порядок после объединения рангов.
4. Если генератор недоступен, чат может вернуть короткий ответ из найденных
   фрагментов.

Деградированный ответ не кэшируется.

## След поиска

Trace, то есть след поиска, нужен не только инженеру. Он показывает пользователю,
как был получен ответ.

В следе поиска сохраняется:

1. Сколько кандидатов пришло из Qdrant.
2. Сколько кандидатов пришло из OpenSearch.
3. Сколько осталось после объединения.
4. Сколько источников попало в ответ.
5. Сколько времени заняли этапы.
6. Был ли резервный режим.
7. Какие цитаты использованы.
8. Раскрывались ли обзорные узлы RAPTOR.

## История

История чата хранится в Postgres.

Когда пользователь открывает старый диалог, модель не вызывается повторно.
Интерфейс показывает сохраненный вопрос, ответ, источники и след поиска.

Это важно для воспроизводимости. Новый корпус или новая модель могут дать другой
ответ, но история показывает результат, который пользователь получил тогда.

## Кэш

Answer cache в Valkey ускоряет повторные одинаковые запросы.

Ключ кэша учитывает:

1. Владельца или общий корпус.
2. Версию корпуса.
3. Текст вопроса.
4. Промпт генерации.

Когда индексируется новый документ, версия корпуса меняется. Старый кэш больше
не используется.

## Настройки

Основные переменные:

1. `RAG_TOP_K`: сколько источников вернуть по умолчанию.
2. `RAG_RETRIEVAL_MULT`: насколько расширить первичный поиск.
3. `RAG_RETRIEVAL_MIN`: минимальная глубина первичного поиска.
4. `RAG_MAX_PER_DOC`: максимум фрагментов из одного документа.
5. `RAG_SCORE_FLOOR`: порог слабой доказательной базы.
6. `RERANKER_URL`: адрес реранкера.
7. `VLLM_URL`: адрес генератора.
8. `RAPTOR_MODE`: режим обзорных узлов.
9. `REGISTRATION_ADMIN_ONLY`: запрет публичной регистрации.
