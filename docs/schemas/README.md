# JSON-схемы

Папка хранит JSON Schema для доменных объектов.

Схемы нужны в трех местах:

1. Структурированный ответ LLM.
2. Проверка API-запросов и ответов.
3. Сопоставление с таблицами `db-service`.

## Гипотеза

Файл: `hypothesis/v1/hypothesis.schema.json`.

Это основная схема результата Фабрики гипотез.

Она описывает:

1. Название и формулировку гипотезы.
2. Метод появления гипотезы.
3. Статус.
4. KPI и кластеры.
5. Уровень готовности.
6. Оценки новизны, риска, ценности и уверенности.
7. План проверки.
8. Доказательства.
9. Происхождение генерации.
10. Итоговый рейтинг.

## Что заполняет сервер

При генерации модель не должна заполнять серверные поля:

1. `id`.
2. `owner_id`.
3. `run_id`.
4. `composite_score`.
5. `created_at`.
6. `updated_at`.
7. Большую часть `generation`.

Сервер заполняет эти поля сам.

## Где хранится

1. Простые поля попадают в таблицу `hypotheses`.
2. `trl` и `assessment` хранятся как JSONB и частично проецируются в колонки.
3. `domain` и `tags` используются для фильтров.
4. `verification`, `quantitative_parameters`, `application_potential`,
   `drivers` и `problem_addressed` хранятся в `detail`.
5. `generation` хранится в одноименном JSONB-поле.
6. `evidence` хранится в строках `hypothesis_evidence`.
7. `composite_score` считает scoring stage.

## Правило изменения

Меняйте схему, SQL-ограничения и код чтения вместе.

Особенно это касается enum-полей:

1. `method`.
2. `status`.
3. `evidence.stance`.
4. `domain.source_type`.
5. `risk.level`.
6. `quantitative_parameters.role`.
