# discovery-worker

`discovery-worker` создает дополнительные гипотезы без ручного запуска.

В текущем compose по умолчанию включен режим `DISCOVERY_MODE=transfer`.

## Режим transfer

Режим transfer переносит сильную гипотезу с одного KPI на близкий KPI, если у
второго KPI не хватает покрытия.

Система:

1. Сравнивает KPI по смысловым векторам.
2. Ищет сильные гипотезы у близкой цели.
3. Адаптирует гипотезу под новую цель через LLM.
4. Проверяет источники и похожесть на уже существующие тексты.
5. Публикует результат на доску.

Основные настройки:

1. `TRANSFER_MIN_SIM`: минимальная близость целей.
2. `TRANSFER_PER_PAIR`: сколько переносов делать на пару целей.
3. `TRANSFER_COVERAGE_FULL`: когда цель считается достаточно покрытой.

## Режим bridge

Режим bridge использует `graph-compute`.

Система:

1. Ждет обновления версии корпуса в Valkey.
2. Вызывает `ScoreBridges`.
3. Получает пары тем и документы-посредники.
4. Просит LLM сформулировать проверяемую гипотезу.
5. Проверяет конвергенцию, источники и новизну.
6. Публикует гипотезу через API.

## Защитные проверки

Гипотеза публикуется только после проверок:

1. Есть реальные источники.
2. Идея не повторяет уже существующий текст корпуса.
3. В режиме bridge есть несколько документов-посредников.
4. Проверка новизны не строится на самооценке LLM.

Если `VLLM_URL` пустой, воркер не должен создавать текстовые гипотезы.

## Основные настройки

1. `DISCOVERY_MODE`: `transfer` или `bridge`.
2. `RAG_GW`: адрес API.
3. `RAG_USER` и `RAG_PASS`: пользователь для публикации.
4. `VALKEY_URL`: адрес Valkey.
5. `EPOCH_KEY`: ключ версии корпуса.
6. `LAST_KEY`: последняя обработанная версия.
7. `GRAPH_COMPUTE_ADDR`: адрес `graph-compute`.
8. `NOVELTY_MAX_SIM`: максимум похожести на существующий текст.
9. `DISCOVERY_LIMIT`: лимит гипотез за запуск.
10. `AUTO_DISCOVERY`: главный выключатель.
11. `EMBEDDINGS_URL` и `EMBEDDINGS_MODEL`: модель для проверки новизны.
12. `VLLM_URL`, `VLLM_MODEL`, `VLLM_API_KEY`: генератор.

